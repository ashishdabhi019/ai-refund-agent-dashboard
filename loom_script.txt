PART 2: Live Demo - Standard Refund (1:00 - 2:30)
-------------------------------------------------------
[Screen: Dashboard UI]

"Let's start with a standard 'happy path' refund."
*Action: Click the Quick Test Scenario pill for "ORD001 Damaged"*
*Action: Wait for the agent to process and generate the Approved receipt card.*

"Here, the customer requested a refund for order ORD001. The agent successfully processed it and generated an approval receipt. 

If you look at the CRM Database on the right, you'll see that order ORD001 has instantly updated its status to 'Refunded' in real-time."

*Action: Click the "Reasoning Logs" button in the top right header to open the sidebar*

"If we open the Reasoning Logs sidebar, we can see exactly how the agent handled this. It followed a strict sequence:
1. Called `lookup_order` to verify the order exists.
2. Called `check_refund_eligibility` to validate the policy rules (it saw the item was damaged and within the 45-day Gold tier window).
3. Called `approve_refund` to finalize it.
4. Sent the final response to the customer."

PART 5: Code Tour & Architecture (5:00 - 7:30)
-------------------------------------------------------
[Screen: Switch to your IDE (VS Code / Cursor)]

"Let's look under the hood at the repository architecture. The app is split into a Vanilla HTML/JS frontend and a FastAPI Python backend."

1. Agent Loop & Tool Orchestration:
*Action: Open `backend/agent.py` and `backend/tools.py`*
"In `agent.py`, I've implemented a raw function-calling agent loop. The agent is forced to use tools deterministically. It cannot blindly approve a refund.
In `tools.py`, you can see the `check_refund_eligibility` tool. I've moved the policy enforcement into code rather than relying entirely on the LLM's prompt. This ensures 100% deterministic policy enforcement. The LLM simply orchestrates the data gathering."

2. Voice Stream Handling:
*Action: Open `frontend/app.html` and scroll to the `startVoiceRecognition()` and `speakText()` functions (around line 1400)*
"On the frontend, the voice stream is handled via the native `webkitSpeechRecognition` API. When the user finishes speaking, the transcript is sent to the `/chat` endpoint. When the response returns, `window.speechSynthesis` reads the text back using a neural voice."

3. CRM & Policy Data:
*Action: Show `backend/crm_data.py` and `backend/refund_policy.py`*
"Here is the mock CRM database with 30 diverse profiles, and the structured strict refund policy document."

PART 6: Reasoning Logs & Failure Handling (7:30 - 8:30)
-------------------------------------------------------
[Screen: Back to Dashboard UI, Reasoning Logs Sidebar open]

"Returning to the Reasoning Logs sidebar. This admin panel acts as our terminal trace. 
If the LLM hallucinates a tool call, or misses a required parameter, the backend catches the error, wraps it in a TOOL_RESULT error message, and feeds it back into the loop."

*Action: In the IDE, briefly show `agent.py`'s try/except block around tool execution.*
"The loop has a `max_iterations` safeguard to prevent infinite loops. If it hits an error, the agent sees the error string and attempts a retry, ensuring robustness."
